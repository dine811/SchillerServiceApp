# Admin dashboard legacy modules (Next.js + Spring Boot)

This document describes the five links from `admindashboard.jsp` (lines 104–118) that were migrated from JSP/servlets to the new stack.

## Flyway note (pre-existing tables)

If PostgreSQL already had `nonsaleablemaster`, `birmaster`, `partnumber_entry`, or `sparepart_master` from an older or partial definition, `CREATE TABLE IF NOT EXISTS` does nothing and indexes on missing columns (e.g. `final_status`) fail. Migrations **V12–V15** therefore include `ALTER TABLE … ADD COLUMN IF NOT EXISTS …` for every column so the schema matches the JPA entities and legacy MySQL dumps.

If a migration **failed** after you changed the script, Flyway may record a checksum mismatch. Options: run `flyway repair` (CLI), or delete the failed row from `flyway_schema_history` for that version (only in dev), then restart the app.

**Primary key `id` / `spare_id`:** If `birmaster`, `nonsaleablemaster`, `partnumber_entry`, or `sparepart_master` existed before Flyway without the surrogate key column (only `ALTER … ADD COLUMN` backfills from V12–V15), Hibernate validation can fail (e.g. `missing column [id]` on `birmaster`). **V16__birmaster_nonsaleable_id_column.sql** adds `id` (or `spare_id` for `sparepart_master`) as `GENERATED BY DEFAULT AS IDENTITY` and a primary key when missing (same approach as **V10** for `prfobmaster`), and optionally renames `bir_id` / `nonsaleable_id` to `id` if present.

**Date column types:** If dates were stored as `VARCHAR` (imports / old DDL), Hibernate reports `wrong column type … found [varchar], expecting [date]`. **V17__birmaster_nonsaleable_date_column_types.sql** converts those columns to `DATE` (and `current_date` to `TIMESTAMP`) when the physical type is still text-like.

## 1. Non salable list — `nonSaleAdminList.jsp`

- **Data:** MySQL table `nonsaleablemaster` (PostgreSQL: Flyway `V12__nonsaleablemaster.sql`).
- **Legacy behaviour:** DataTables server-side JSON from `NonSaleAdmin_PageController` — **all rows with `final_status = Pending`** (no division filter on the count/query).
- **Export:** `ExportNonSale_AdminController` → `ExportNonSaleDao.NonSaleAdmin(from, to, status, division, …)` — filters by `entry_date` between from/to, `final_status`, and division (unless division is `"1"` = all).
- **New API:** `GET /api/nonsaleable/admin/pending`, `GET /api/nonsaleable/admin/export`, `DELETE /api/nonsaleable/{id}`.
- **UI:** `/dashboard/non-salable-admin` — list, search, export, delete.

## 2. Salables — `SalablesList.jsp`

- **Same table** as non-salable: `nonsaleablemaster` with **`final_status = Completed`**.
- **Legacy:** `SalablesPage_Controller` — admin/FQC see all completed; other roles filter by the user’s division from session.
- **New API:** `GET /api/nonsaleable/salables?division=…` — optional division (empty = all, matching admin view); **engineers should select their division** in the UI until auth is wired.
- **Export:** Same export endpoint with `status=Completed`.
- **UI:** `/dashboard/salables`.

## 3. BIR list — `BIRAdminList.jsp`

- **Data:** `birmaster` (`V13__birmaster.sql`).
- **Legacy:** `BIRAdmin_PgeController` — **all rows with `final_status` pending** (MySQL case-insensitive; new API uses `lower(trim(final_status)) = 'pending'`).
- **Export:** `ExportBIR_AdminController` → `Export_BIRDao.BIRAdmin` — filters by `fqc_in_date` between from/to, `final_status`, division optional (`"1"` = all).
- **New API:** `GET /api/bir/admin/pending`, `GET /api/bir/admin/export`, `DELETE /api/bir/{id}`.
- **UI:** `/dashboard/bir-admin`.

## 4. Closed BIR — `ClosedBIRList_admin.jsp`

- **Same table** `birmaster` with **`final_status = Completed`**.
- **Legacy:** `BIRClosedPge_Controller` — admin / ViceChancellor / FQC / ProductSupport see all completed; others restricted by division.
- **New API:** `GET /api/bir/admin/completed?division=…` — optional division filter.
- **Export:** Same BIR export with `status=Completed`.
- **UI:** `/dashboard/bir-closed-admin`.

## 5. Spares entry — `partNumberForm.jsp`

- **Legacy:** POST to `PartEntry` servlet → `partnumber_entry` (`Part_number`, `brd_Name`, `compatible_models`, `cost`).
- **Note:** Distinct from catalogue table `sparepart_master` (used elsewhere for lookups). Flyway `V14__partnumber_entry.sql`, `V15__sparepart_master.sql` adds the missing `sparepart_master` table for the existing `SparePart` JPA entity.
- **New API:** `POST /api/partnumber-entry` with JSON body `{ partNumber, brdName, compatibleModels, cost }`.
- **UI:** `/dashboard/spares-entry`.

## Cross-cutting

- **Model names:** List DTOs resolve `model` id via `model` table when the value is numeric (same idea as legacy `ModelDao.getModelname`).
- **Excel exports:** Implemented with Apache POI (`.xlsx`), aligned with legacy column sets where practical.
- **Edit flows:** Legacy JSPs used `NonSaleControler` / `BIRController` for update; **not** reimplemented here — only list, export, delete, and spares create.
